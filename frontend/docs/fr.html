1. Vue d'ensemble

1.1 Ce que fait Helios

Helios est une carte personnalisée Lovelace pour Home Assistant qui visualise en temps réel les conditions solaires au domicile de l'utilisateur sur une carte 3D MapLibre. La carte agrège :

  • L'arc solaire sur 24 h, le disque solaire et l'irradiance live
  • La couverture nuageuse (modèle Open-Meteo multi-source)
  • Les ombres portées au sol par les bâtiments ET la végétation (quand un provider LiDAR national couvre le domicile)
  • La production PV instantanée (capteur HA), avec une prédiction auto-corrigeante
  • L'état de la batterie domestique (capteurs HA multi-banks)
  • Une timeline scrubbable sur 5 jours (J-2 à J+2) qui pilote l'ensemble de la scène

L'utilisateur ne fournit aucune clé d'API : la météo vient d'Open-Meteo, le fond de carte d'OpenFreeMap, les rasters LiDAR des programmes open-data nationaux.

1.2 Philosophie de conception

Quatre contraintes structurent l'ensemble du code :

  1. Zéro friction utilisateur. Pas de configuration obligatoire, pas de clé d'API, pas de compte. Tout doit fonctionner avec un type: custom:helios-card sec dans le YAML.
  2. Transparence. Tout est calculé côté client dans le navigateur, l'utilisateur peut auditer chaque étape de la prédiction. Pas de black box serveur.
  3. Précision croissante via apprentissage local. Le modèle analytique fait office de baseline. Une calibration 5-jours (scalaire) puis une carte d'ombrage adaptative (par-cellule) apprennent les biais propres à l'install au fil des jours.
  4. Pas de dépendance commerciale. OpenFreeMap (vector tiles), Open-Meteo (météo), MapLibre (rendu), GeoTIFF.js (LiDAR) : tout est gratuit, sans rate limit, sans compte.

2. Architecture générale

2.1 Le split card / engine

Le projet repose sur deux orchestrateurs top-level :

  • src/helios-card.ts : un LitElement qui gère le cycle de vie côté Home Assistant, écoute les changements de config et de hass.states, et oriente le rendu DOM / SVG. Il NE TOUCHE PAS à MapLibre directement.
  • src/helios-engine.ts : une classe HeliosEngine qui possède l'instance MapLibre, les sources, les layers, les calculs de projection, les fetches LiDAR. Il NE CONNAÎT PAS Lit ni Home Assistant.

L'interface entre les deux passe par des callbacks installés par la card sur l'engine (engine.onShadowComputeStart = () => ..., engine.onMapTransform = () => ...) et par des getters côté engine (engine.projectHomeLabelLayout(), engine.getLidarRaster(), etc.).

Pourquoi ce split : MapLibre + WebGL est lourd et stateful, son cycle de vie ne correspond pas à celui d'un élément Lit (qui peut être détaché / réattaché à n'importe quel moment par les layouts HA Masonry / Sections). La séparation permet à la card de mourir et ressusciter sans toucher à l'engine, et inversement.

2.2 Le pattern "host structurel"

Chaque sous-système dans card/*.ts ou engine/*.ts est une collection de fonctions libres qui prennent un host en paramètre. Le host est une interface TypeScript déclarée dans le fichier lui-même, qui décrit la surface dont la fonction a besoin sur l'objet appelant.

Exemple, depuis src/card/battery.ts:32-45 :

export interface BatteryHost
{
    readonly config:     HeliosConfig | undefined;
    readonly hass:       any;
    readonly _timeRange: { start: Date; end: Date } | null;

    _batterySoc:          number | null;
    _batteryPower:        number | null;
    _batteryPowerUnit:    string;
    _batterySocHistory:   BatteryHistory | null;
    _batteryPowerHistory: BatteryHistory | null;
    _batteryFetchKey:     string;
    _batteryFetching:     boolean;
}

refreshBattery(host: BatteryHost): void mute les champs _battery* du host directement. Lit voit les assignations grâce aux setters installés par @state() sur helios-card.ts et planifie un re-render automatiquement.

Avantages :

  • Pas de service classes, pas de DI container
  • Chaque module reste testable isolément
  • Le typage TypeScript empêche un module de toucher à un champ qu'il n'a pas demandé
  • L'arborescence des dépendances est explicite

3.2 Patterns de fichiers

  • card/*.ts : fonctions libres + XxxHost interface
  • engine/*.ts : un mélange (la pipeline LiDAR est procédurale, le layer LiDAR-View est une classe parce que MapLibre exige une classe avec onAdd / render / onRemove)
  • Chaque fichier commence par un commentaire bloc qui explique le rôle du module, les invariants, et le pattern utilisé. Lisez-les d'abord, ils sont denses mais à jour.
  • Pas de docstring sur les fonctions privées triviales. Les commentaires existants documentent les choix non-évidents (pourquoi ce seuil, pourquoi cet ordre, pourquoi ce repli sur erreur).

4. Cycle de vie d'une carte

4.1 Bootstrap

L'élément Lit <helios-card> est instancié par Home Assistant quand le YAML déclare type: custom:helios-card. La séquence d'init :

  1. setConfig(config) est appelé par HA, stocke la config dans this.config (déclencheur Lit).
  2. connectedCallback() (helios-card.ts) installe les listeners globaux (visibility observer, resize, hass-update).
  3. firstUpdated() ne fait RIEN d'engine, on attend que hass soit poussé.
  4. Quand hass arrive, init.ts:initEngine(host) lance un requestAnimationFrame après un debounce de 500 ms.
  5. Le 500 ms évite que les preview cards éphémères de l'éditeur HA (la card mini qui apparaît dans le picker) spawnent un MapLibre et claim un contexte WebGL pour rien. Voir src/card/init.ts:260 et le commentaire au-dessus.
  6. initEngineNow() crée un new HeliosEngine(), l'attache au <div id="map-container"> rendu par Lit.
  7. L'engine fait son _initMap(), _loadStyle(), attend style.load, puis _onStyleLoad() ajoute toutes les sources + layers Helios sur la carte.
  8. À partir de là, _refreshShadowsAndAtmosphere() tourne sur un timer de 30 s, et la timeline + le scrub deviennent fonctionnels.

4.2 Updates Home Assistant

Quand hass est repoussé par HA :

  • set hass(h) (setter sur la card) déclenche refreshPv(this), refreshBattery(this), refreshSolarRadiation(this).
  • Si la config a changé (comparée par _lastConfigSig), updateConfig() est appelé sur l'engine pour propager.
  • Lit re-render automatiquement si un @state a été touché.

4.3 Teardown

disconnectedCallback() (helios-card.ts) :

  • Annule les RAFs (_lidarFadeRaf, _shadingDomeFadeRaf, _dashCountUpRaf, _autoRotateRaf)
  • Disconnect l'IntersectionObserver
  • Clear l'interval d'horloge
  • Schedule un cleanup différé de l'engine après 1500 ms (_pendingCleanupTimer), parce que les layouts Masonry détachent / rattachent la card fréquemment. Si la card est rattachée avant 1500 ms, l'engine survit et reprend.

HeliosEngine.cleanup() :

  • Cancel tous les AbortControllers
  • Cancel tous les rAF / idle handles (_exposureChunkRaf, _exposureIdleHandle)
  • Supprime explicitement chaque layer + source MapLibre (le removeLayer('helios-lidar-view') est crucial pour libérer les buffers GPU du custom layer)
  • Désenregistre tous les map.off()
  • Force WEBGL_lose_context.loseContext() pour relâcher le slot WebGL
  • Sort de _liveEngines (le set qui force l'éviction au-delà d'un cap)
Toute fuite mémoire dans le projet vient d'un oubli dans cleanup(). Lire ce code une fois.

5. Le moteur (helios-engine.ts)

HeliosEngine est la plus grosse classe du projet (~4300 lignes). Elle est volontairement monolithique pour les raisons suivantes :

  • MapLibre est stateful : chaque méthode doit voir l'état complet de la carte (zoom, pitch, bearing, sources, layers, etc.).
  • Splitter en sous-classes nécessiterait soit du multi-héritage soit une chaîne d'appels qui obscurcit ce qui se passe vraiment.

Les méthodes publiques exposées à la card :

// Bootstrap
constructor(opts)
public onMapTransform?: () => void                 // installé par la card
public onShadowComputeStart?: () => void           // pareil
public onShadowComputeEnd?: () => void

// Configuration
public updateConfig(cfg, hass): void               // propagé par la card
public setHome(lat, lon): void

// Projections (la card lit ces snapshots pour ses overlays SVG)
public projectHomeLabelLayout(): {...}
public projectHomeFootprints(): Array<...>
public projectSunScene(): SunScene
public projectCloudScene(): CloudScene

// Modes
public setDetailMode(on): void
public setLidarViewActive(on): void
public setLidarViewOpacity(opacity): void
public setLidarViewFadeAlpha(alpha): void
public setAutoRotateEnabled(on): void

// LiDAR
public getLidarRaster(): LidarRaster | null
public getActiveLidarSourceId(): string | null

// Lifecycle
public cleanup(): void

Les champs internes critiques :

  • this.map : l'instance maplibregl.Map
  • this._lidarRaster / this._lidarShadowFeatures : raster brut + features dérivées (polygones d'ombre)
  • this._lidarShadowKey / _lidarShadowFailedKey / _lidarShadowBackoffUntil : cache key + backoff sur les fetches LiDAR (voir le système LiDAR)
  • this._lidarViewLayer : instance du custom WebGL layer
  • this._weatherCache : cache des fetches Open-Meteo
  • this._buildingsData : features du fond de carte filtrées au rayon

Le timer principal _skyTimer tourne toutes les 30 secondes : _refreshShadowsAndAtmosphere() recalcule la position du soleil, et si l'altitude a bougé de plus de 0.5°, redessine les shadows + met à jour le night-shade overlay + schedule un nouveau calcul d'exposure LiDAR-View.

6. Le rendu (helios-card.ts + card/*.ts)

helios-card.ts:render() est une méthode unique de plus de 1000 lignes qui assemble le template Lit. Sa structure :

<ha-card class="..."> // classes : theme + mode flags
  <div id="map-container"></div>            // hôte MapLibre
  ${timeline-strip si configuré}
  ${overlay-top-left : clock + scrub-return}
  ${overlay-top-right : mode-bar : Layer / LiDAR / Dome}
  ${cloud-svg + cloud-leader + cloud-pct-label}
  ${solar-svg + solar-pct-label : arc + disque + halo}
  ${pv-home-anchor-svg + pv-pct-label}
  ${battery-leader-svg + battery-pct-label x 2}
  ${home-silhouette-svg + home-glow-svg + home-hitbox}
  ${incidence ray}
  ${dashboard panel si _detailMode}
  ${shading dome overlay si _shadingDomeMode}
  ${LiDAR-View opacity picker si _lidarViewMode}
</ha-card>

Cette méthode est appelée par Lit à CHAQUE changement d'un @state field. Tous les frames de scrub (touch / mouse drag) déclenchent un re-render complet. Voir pièges connus pour les implications perf.

Sous-modules de rendu

  • card/overlays.ts : refreshOverlays(host) est appelé par l'engine via onMapTransform. Il appelle engine.projectHomeLabelLayout() (et autres), stocke les résultats dans des @state fields sur la card, ce qui déclenche Lit. C'est le pont entre les transformations MapLibre et le rendu SVG du card.
  • card/charts.ts : la timeline (irradiance + cloud + PV chart optionnel) + le day-strip avec totaux kWh par jour. Calcule effectiveForecastRatio (calibration + shading map) à chaque point.
  • card/dashboard.ts : le panneau de détail rendu quand _detailMode === true. Implémente le counter-up animation 700 ms via un rAF loop self-terminant.
  • card/shadingDome.ts : l'overlay hémisphérique de la shading map (mode dôme). SVG-heavy, re-projeté à chaque transform.

7. Sous-systèmes du moteur (engine/*.ts)

7.1 engine/sun.ts

Implémente la chaîne de calcul solaire. Trois fonctions exposées :

  • getSunPosition(date, lat, lon) : retourne {altitude, azimuth} en degrés. Math NOAA simplifiée : déclinaison + équation du temps + angle horaire. Validée à 0.30° d'erreur moyenne sur l'altitude et 0.36° sur l'azimut contre 376 points de référence.
  • computePvPower(config, time, lat, lon, cloud, ctx) : préprediction PV pour UN array (tilt, azimuth, etc.). Chaîne complète : position solaire → GHI ciel-clair Haurwitz → atténuation Kasten-Czeplak → décomposition beam/diffus → transposition Liu-Jordan → dérating Sandia NOCT → raycast LiDAR si raster disponible. Retourne un pourcentage [0,1] de la STC.
  • computePvPowerWeighted(config, time, lat, lon, cloud, ctx) : agrège les pv-arrays (multi-orientations) en pondérant par peak-kwp ou share.

7.2 engine/weather.ts

Fetch multi-modèle Open-Meteo + cache + back-off 429. Pour chaque fetch, query :

  • Un modèle global (ECMWF IFS 0.25°)
  • Le meilleur modèle national/régional couvrant le point (AROME-France, UKMO UK, DWD ICON-D2, NOAA HRRR, BOM ACCESS-G, etc.)

Fusion par médiane par-timestep, ce qui absorbe les outliers d'un modèle isolé. Cache LRU 30 minutes par signature (lat, lon, forecastDays, pastDays, modelSet). Seul le premier hit après 30 min refait un réseau.

7.3 engine/buildings.ts

Fetch OpenFreeMap planet vector tiles autour du domicile, décode via @mapbox/vector-tile, splitte les MultiPolygons qui groupent des bâtiments indépendants, filtre au rayon building-radius. Sépare les bâtiments en deux ensembles :

  • home cluster : le bâtiment au domicile + ceux dans building-cluster-radius (vérandas, garages collés), rendu full opacity.
  • surroundings : les autres dans building-radius, rendu à building-opacity.

Les deux deviennent des MapLibre fill-extrusion layers avec fill-extrusion-height lu sur la feature.

7.4 engine/shadows.ts + shadow-raster.ts

Projette les polygones de bâtiments AU SOL selon la position du soleil (extrusion ombre). Pour chaque bâtiment :

  1. Pour chaque vertex, projette (x, y, height) → (x', y') au sol en suivant le rayon solaire inverse.
  2. Concatène les vertices du bâtiment + les vertices projetés.
  3. Calcule l'enveloppe convexe Sutherland-Hodgman du tout.

Les polygones d'ombre sont rastérisés en noir sur un canvas offscreen, puis ce canvas devient une image source MapLibre. Un raster layer en lit avec raster-opacity: shadow-opacity. Cette approche évite la saturation alpha-compositing quand des bâtiments se chevauchent.

Quand un provider LiDAR couvre le domicile, on n'utilise pas du tout shadows.ts. À la place, la pipeline LiDAR produit ses propres polygones via flood-fill.

7.5 engine/lighting.ts

Modulation jour/nuit. Calcule l'opacité du night-shade overlay (smoothstep selon altitude solaire), la teinte ambré des bâtiments dans la fenêtre sunrise / sunset, et la couleur du sun-arc en fonction de la phase. Appelé depuis _refreshShadowsAndAtmosphere à chaque tick sky-timer.

7.6 engine/auto-rotate.ts

Boucle rAF d'orbite caméra. Self-terminating quand auto-rotate-enabled est false OU une interaction utilisateur récente pause l'orbite. Vitesse 1°/s dans le sens contraire au soleil.

7.7 engine/detail-mode.ts

Quand le détail s'active, anime la caméra via smoothstep (zoom de 0.8 à 1.4 sur 800 ms, pitch vers ~50°). Token rAF stocké dans _detailDiveRaf, annulé sur sortie.

7.8 engine/lidar-view-layer.ts

Custom layer WebGL implémentant CustomLayerInterface MapLibre. Trois passes de draw : Points (drawArrays), Wireframe (drawElements connectant les voisins), et Irradiance fill (triangulation avec exposure per-vertex).

7.9 engine/pv-shading.ts

  • isPanelShaded(raster, lat, lon, height, sunAlt, sunAz) : raycast depuis un point panneau vers le soleil contre le nDSM, retourne true si bloqué. Utilisé pour zéroer le beam direct sur une string ombragée.
  • computeLidarCellExposureRows(...) : raycast par-cellule pour la carte d'irradiance, écrit 0 (ombre) ou 255 (lit) dans le buffer de sortie.

Les deux raycasts utilisent un step de 2 m et une portée max de 200 m. Le step bilinéaire échantillonne le nDSM (et le DTM si présent) pour comparer la hauteur du rayon solaire à celle des obstacles.

7.10 engine/shadingMap.ts

Le store + lookup de la carte d'ombrage adaptative (Grille polaire de 1200 cellules : 24 azimuts × 10 altitudes × 5 couvertures nuageuses). Chaque cellule stocke (ratioSum, countWeight, lastUpdatedMs). Persisté en localStorage.

8. La prédiction PV : chaîne de calcul

Cette section décrit computePvPower pas-à-pas. C'est le cœur de la prédiction théorique et analytique.

8.1 Position du soleil

Pour un instant $t$ à $(\text{lat}, \text{lon})$, getSunPosition calcule :

  1. Jour julien fractionnaire depuis l'epoch GPS.
  2. Déclinaison solaire $\delta$ : $$ \delta = 23.45^\circ \times \sin\left(360^\circ \times \frac{284 + N}{365}\right) $$ où $N$ est le jour de l'année.
  3. Équation du temps $\text{EOT}$ (en minutes) : $$ B = 360^\circ \times \frac{N - 81}{365} $$ $$ \text{EOT} = 9.87 \times \sin(2B) - 7.53 \times \cos(B) - 1.5 \times \sin(B) $$
  4. Heure solaire vraie : $\text{TST} = \text{UTC} + \frac{\text{lon}}{15} + \frac{\text{EOT}}{60}$
  5. Angle horaire : $\text{HA} = 15^\circ \times (\text{TST} - 12)$
  6. Altitude solaire : $$ \sin(\text{alt}) = \sin(\text{lat}) \times \sin(\delta) + \cos(\text{lat}) \times \cos(\delta) \times \cos(\text{HA}) $$
  7. Azimut solaire (clockwise depuis le nord) : $$ \sin(\text{az}) = \frac{-\cos(\delta) \times \sin(\text{HA})}{\cos(\text{alt})} $$ $$ \cos(\text{az}) = \frac{\sin(\delta) - \sin(\text{alt}) \times \sin(\text{lat})}{\cos(\text{alt}) \times \cos(\text{lat})} $$ $$ \text{az} = \text{atan2}(\sin(\text{az}), \cos(\text{az})) $$

8.2 Irradiance ciel-clair (Haurwitz)

Pour une altitude solaire $\text{alt} > 0$, le GHI ciel-clair ($\text{W/m}^2$) est calculé par :

$$ \text{GHI\_clear} = 1098 \times \cos(z) \times \exp\left(\frac{-0.057}{\cos(z)}\right) $$

où $z = 90^\circ - \text{alt}$ est l'angle zénithal.

8.3 Atténuation nuageuse (Kasten-Czeplak)

$$ \text{GHI\_actual} = \text{GHI\_clear} \times \left(1 - 0.75 \times \left(\frac{\text{cloud}}{100}\right)^{3.4}\right) $$

8.4 Décomposition beam / diffus

On sépare le GHI horizontal en composante directe (beam) et diffuse :

directFraction = (kCloud - 0.25) / 0.75 * 0.85
diffuse        = (1 - directFraction) * GHI_actual
beam           = directFraction * GHI_actual
Note : cette décomposition ignore la masse d'air. Pour un soleil très bas, le modèle peut sur-estimer la part directe. C'est compensé downstream par la calibration 5 jours.

8.5 Transposition Liu-Jordan (POA)

Conversion du GHI horizontal en POA (Plane Of Array) selon l'inclinaison du panneau :

  1. Cosinus de l'angle d'incidence $\cos\theta$ : $$ d\text{Az} = \text{sun.azimuth} - \text{panel.azimuth} $$ $$ \cos\theta = \sin(\text{alt}) \times \cos(\text{tilt}) + \cos(\text{alt}) \times \sin(\text{tilt}) \times \cos(d\text{Az}) $$
  2. Ratio beam ($R_b$) : $$ R_b = \frac{\max(0, \cos\theta)}{\max(\sin(5^\circ), \sin(\text{alt}))} $$ $$ \text{beamPOA} = \text{beam} \times R_b $$
  3. Diffus isotropique : $\text{diffusePOA} = \text{diffuse} \times \frac{1 + \cos(\text{tilt})}{2}$
  4. Réflectance sol (albedo = 0.20) : $\text{groundPOA} = \text{GHI\_actual} \times 0.20 \times \frac{1 - \cos(\text{tilt})}{2}$
  5. $\text{POA Total} = \text{beamPOA} + \text{diffusePOA} + \text{groundPOA}$

8.6 Dérating thermique Sandia NOCT

Chute de rendement liée à la température :

$$ T_{\text{cell}} = T_{\text{air}} + \frac{\text{NOCT} - 20}{800} \times \text{GHI} - 1.5 \times \text{wind} $$ $$ \text{factor} = 1 + \gamma_{\text{pmp}} \times (T_{\text{cell}} - 25) $$

Avec $\text{NOCT} = 45^\circ\text{C}$ et $\gamma_{\text{pmp}} = -0.0040/^\circ\text{C}$ (mono-Si typique).

8.7 Raycast LiDAR (ombrage par obstacle)

Pour un pas de 2 m à 200 m, on échantillonne le nDSM (hauteur d'obstacle) et le DTM (pente terrain). Si $\text{obstacleZ} > \text{rayZ}$, l'ombre est détectée : on zéro le beam direct mais on conserve l'apport du diffus et du sol (ground). Le panneau tombe à ~25-30% au lieu de 0.

8.8 Agrégation multi-array

Pondération des différents calques d'orientation (pv-arrays) par leur puissance crête (peak-kwp) ou leur part (share), suivie du plafonnement par l'onduleur (pv-inverter-max-kw).

9. Le système LiDAR

9.1 Registry de providers

Le fichier src/engine/lidar.ts expose le tableau LIDAR_SOURCES contenant les providers actifs (10 actuellement enregistrés : fr.ts (IGN HD), uk.ts, es.ts, nl.ts, no.ts, de-nrw.ts, de-bb-be.ts, pl.ts, ca.ts, us-vt.ts). Quatre sont actuellement dormants (attente de qualité DSM-DTM).

9.2 Pipeline partagée (flood-fill + convex hull)

processNdsmToShadows gère la transformation du raster en polygones exploitables :

  1. Threshold + crop circulaire autour du domicile.
  2. Flood-fill 8-connecté avec cap à 80 cellules par clump (permet de découper les barres d'immeubles ou haies d'arbres continues).
  3. Calcul de l'enveloppe convexe (Convex Hull) par clump pour générer des Features GeoJSON identiques à OpenFreeMap.

9.3 Format GeoTIFF / nDSM 2-bandes

  • 1-bande : Float32 correspondant à la hauteur brute de l'obstacle au-dessus du sol.
  • 2-bandes (recommandé) : Bande 1 = nDSM, Bande 2 = DTM (élévation absolue du sol). Permet de gérer fidèlement le relief topographique montant ou descendant.

9.4 La couche WebGL custom

Le traitement s'effectue en 3 passes GPU dans lidar-view-layer.ts : points, irradiance fill (coloration chaud/froid selon l'attribut a_exposure), et wireframe.

Le calcul d'exposure est découpé par paquets de 8 lignes (CHUNK_ROWS = 8) via rAF pour ne jamais bloquer le thread principal (opération transparente < 16 ms).

Gestion des erreurs / backoff : En cas d'erreur réseau/CORS répétée, un mécanisme de backoff exponentiel bloque les requêtes inutiles (60 s → 5 min → 15 min → 30 min → 60 min).

10. Les couches d'apprentissage

10.1 Calibration 5 jours (scalaire)

Calcule le rapport global entre la production réelle observée (historique HA) et la prédiction théorique du modèle sur les 5 derniers jours. Ce ratio, borné à $[0.5, 1.5]$, absorbe les biais statiques (vieillissement des panneaux, salissures, pertes constantes de l'onduleur).

10.2 Carte d'ombrage adaptative (vectorielle)

Mise à jour toutes les 5 minutes dans les bins polaires (1200 cellules). Calcule en continu le ratio réel/prédit par tranche de 30 minutes. Lors de la lecture, les valeurs anciennes subissent une décroissance via une demi-vie de 60 jours.

10.3 Le guard cutoff onduleur

Si la batterie est pleine, l'onduleur bride souvent la production PV. Le trainer exclut ces tranches horaires (min(SoC across banks) >= cutoff) pour éviter d'enregistrer de "fausses ombres" (ombres fantômes).

10.4 Le blend effectiveForecastRatio

La règle de priorité est simple : si la cellule polaire de la Shading Map dispose d'assez de données (seuil de confiance), elle prévaut. Sinon, le moteur applique le ratio scalaire global des 5 jours.

11. La timeline et le scrub temporel

La barre basse combine 3 graphs au format SVG (Production PV, Irradiance + Nuages, Day-strip avec totaux kWh). Le glissement du pointeur (scrub) met à jour la variable d'état réactive _selectedTime, forçant le recalcul complet de la scène 3D (position du soleil et ombres associées) à l'instant ciblé.

12. Le dashboard (mode détail)

Déclenché par un clic sur la maison. La caméra effectue un plongeon (smoothstep zoom + pitch de 50° en 800 ms), l'interface HUD s'estompe au profit du panneau .detail-panel. Les chiffres de production s'animent via une boucle rAF de 700 ms (effet de compteur progressif).

13. Les modes spéciaux (LiDAR-View, dôme)

  • LiDAR-View : Active le nuage de points et estompe le HUD. Le curseur d'opacité modifie une variable simple (plain field) pour éviter de saturer Lit avec 50 re-renders par seconde lors du glissement.
  • Dôme d'ombrage : Représentation SVG hémisphérique du ciel au-dessus de la maison. Colore visuellement chaque case de la Shading Map apprise (Rouge = ombragé/sous-production, Vert = dégagé, Gris = inconnu).

14. Internationalisation (i18n)

Le fichier src/i18n/index.ts centralise le type strict Translations. Le projet supporte 11 langues. Le compilateur tsc bloque le build si une clé ajoutée n'est pas répercutée dans l'intégralité des 11 fichiers de locales.

15. Configuration + éditeur visuel

La source de vérité des paramètres réside dans l'interface HeliosConfig (src/helios-config.ts). L'éditeur visuel UI (src/card/editor.ts) se compose d'un accordéon de 14 sections rétractables. Les tableaux complexes (orientations de panneaux ou banques de batteries) se pilotent via un pattern dynamique d'ajout/suppression de lignes avec un plafond strict fixé à 6 rangées maximum.

16. Conventions et patterns

  • Style : TypeScript strict. Pas de tirets cadratins (—) dans les commentaires pour éviter le style généré par IA. Pas d'historique de version dans le code. Messages de commit standardisés : vX.Y.Z[-suffix]: titre.
  • Host structurel : Modules isolés via des fonctions pures travaillant sur une interface Host partagée. Pas de conteneur d'injection de dépendances.
  • Mémoisation : Les calculs lourds dépendants de la configuration sont mis en cache sur l'identité de l'objet this.config.
  • Ressources GPU / CPU : Nettoyage rigoureux (cleanup) obligatoire des handles rAF, annulation des requêtes fetch via AbortController, et destruction explicite du layer WebGL (removeLayer) sous peine de fuites mémoire critiques, notamment sous iOS Safari.

17. Comment contribuer

17.1 Ajouter un provider LiDAR

Créer un fichier dans src/engine/lidar/providers/[code_pays].ts implémentant l'interface LidarSource (définir la Bounding Box de couverture, les URLs de fetch DSM/DTM, et appeler processNdsmToShadows). Enregistrer ensuite la constante dans la liste LIDAR_SOURCES du fichier src/engine/lidar.ts.

17.2 Ajouter une clé de config

  1. Déclarer la propriété typée unknown dans src/helios-config.ts et définir sa valeur par défaut.
  2. Écrire une fonction de parsing/validation sécurisée.
  3. Ajouter la clé dans le tableau WATCH_KEYS (src/card/init.ts) pour forcer la mise à jour réactive.
  4. Créer l'élément de contrôle HTML/Lit correspondant dans l'éditeur visuel (src/card/editor.ts) et enrichir les dictionnaires i18n.

17.3 Ajouter une langue

Créer la locale dans src/i18n/locales/, traduire l'intégralité des entrées calquées sur le modèle de la structure Translations, puis référencer le fichier dans l'objet LOCALES de src/i18n/index.ts.

18. Pièges connus

  • Lit re-render storm : Le frottement rapide de la timeline sollicite fortement la méthode render(). Les variables annexes comme l'opacité sont volontairement stockées hors de l'état réactif pour préserver les performances.
  • WebGL Buffers Leak : MapLibre détruit mal les contextes WebGL sur mobile. L'appel explicite à map.removeLayer() doit impérativement précéder le déchargement de la carte.
  • Cumulative energy day-boundary : Les deltas de production à cheval sur minuit (ex: sample à 23h58 puis 00h05) faussent les statistiques. La fonction de calibration rejette systématiquement les calculs dont les deux points temporels n'appartiennent pas à la même journée calendaire.
  • Précision Float32 en coordonnées mondiales : Les grands nombres de la projection WebMercator dégradent la précision 32-bit loin de l'équateur. Le Custom Layer applique un décalage (shift) local pour basculer le repère géométrique $(0,0)$ directement sur la maison de l'utilisateur.
